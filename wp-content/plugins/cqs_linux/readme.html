Overview

Version Difference

A.M. Best Version

The A.M. Best Version of the Compulife Internet Engine can show the A.M. Best ratings. This version will only be sold to those people who have provided Compulife with a written agreement from A.M. Best indicating they have permission to use A.M. Best ratings in their comparisons.

Basic Engine

The basic Engine allows comparisons optionally using the health analyzer, up to 6 side by side comparisons, Income Replacement Calculator, and Easy Retirement Calculator.

New Features

This lastest incarnation of the Compulife Internet Engine has several new features.

  • New categories

    The categories selection has been changed to reflect the new categories that Compulife supports including No lapse UL. See NewCategory Syntax for more information on how to use it.

  • The ability to display two modal columns in side by side display.

    This means that you can (for example) show Annual and Monthly premiums side by side in a comparison see How to display a comparison result for more information.

  • Expanded User Controls for application requests for weblinks, emails, and css style options

    See company selection for more information

  • Mini Health Analysis

    The mini health analyzer only includes the basic health analyzer questions. (no Driving, Family Health History or Substance abuse questions). This is in addition to the full health anaylzer in the system

  • Template file overrides

    Any of the template files can be given a new name. This allows you to have special case template files. Any file in the system can be replaced by appending FILE to the name see Template file and how it works for more information.

  • Even and Odd Row markers to assist with CSS

    Comparison results can now have Even and Odd row markers which can be be used along with css to give different visual attributes to the rows

UserLocation

Some time back, we introduced an input variable called UserLocation. If you look in the usmain.html file, you will see it. The default location is in the directory consumer located in COMPLIFE/USER/CONSUMER recommended that you make use this variable so that you can put your *.HTM files in its own directory.In the sample file, we set UserLocation's value to be "USER2".For Linux users, they will need a directory called public_html/cgi-bin/COMPLIFE/USER/USER2/ . For Windows users, they will need a directory called public_html\COMPLIFE\USER\USER2\ . The *.HTM files will go into the USER2/ directory.

There are many benefits to using the UserLocation variable. First off, it is very clean because the*.HTM files will not be lying around in your public_html or your cgi-bin directory. Secondly it allows you to have multiple starting pages. Say you wanted to use a different start page for testing. To do this, copy usmain.html to mytest.html. Edit mytest.html and change value="USER2" to value="MYTEST". Create the../COMPLIFE/USER/MYTEST/ directory. Copy the *.HTM files to the MYTEST directory. Now you can alter the *.HTM files in the MYTEST directory leaving your usmain.html lookup unaffected since it is using the *.HTM files from the USER2 directory.

IMPORTANT: For those people who had been putting the *.HTM in their main public_html(Windows) or public_html/cgi-bin(Linux) directory and who decide they want to the UserLocation input variable. Please make sure that UserLocation is passed through as a hidden variable in the rest of your *.HTM files. I would suggest putting the <input type="hidden" name="UserLocation" value="$userlocation$> line under the <input type="hidden" name="Smoker" value="$origsmoker$"> line that is already there. Grouping it like this makes it easier to read.

Debugging

A problem we have noticed when using template or custom template files, is that it is easy to lose track of which template file you are dealing with. A trick that we have incorperated in our engine is to have the engine showing the filename that it is currently using as a template. The $filename$ keyword. This keyword will only work if a file named "debug" is located in the cgi-bin directory. Otherwise instead of showing the filename being used, it will show "". This is so that instead of having to edit all the pages to remove the debugging information, you only have to remove one file.The contents of the debug file are unimportant, and it can be 0 length. Template files and how it works

Template files tell engine how to format the output of the data. To do that, it has special dollar codes which get replaced by the engine with actual dollar values. All the other information is passed through to the output as is, so you can arrange your output however you want.

For example the line
The faceamount is $faceamount$
in the TEMPLATE.HTM will cause the engine to substitute the actual face amount with the $face$ perhaps giving you
The faceamount is $500,000
This simple method can also be used to pass variables through forms. For example
< input="hidden" name="FaceAmount" value="$face$" >
will pass on the face amount value into the engine on the next submit.

The actual name of the template file can be changed in this method as well.

How to display the comparison result

TEMPLATE.HTM is the main template file used to display the comparison results. To display the comparison result table you will need to define its starting point with $prodline=50$. You will also need to define it sending point with $/prodline$. The 50 represents the maximum number of results to display. In between the $prodline=50$ and $/prodline$ will be the prodline definition (the html code that will represent how you want the table to look). The code between the starting and ending prodline markers can have a maximum of 4000 characters. There will be special dollar codes that you can use within the prodline definition which will translate into information for the products that are illustrated. Dollar codes which are available for use within the prodline definition are:

Dollar Codes Description

$company$

The company name

$product$

The product name

$healthcat$

The health category for the product. for example: Super Preferred Non-Tobacco

$PremiumAnnual$

The product's Annual premium

$Premium$

The product's premium (Annual,Monthly,Quarterly or Semi-Annual selected by Input variable ModeUsed)

$rgpfpp$

Regular, Regular Plus, Preferred and Preferred Plus (Rg, R+, Pf, P+)

$guar$

becomes 'gtd' if the premium is guaranteed else becomes '**'

$policyfee$

The annual policy fee of the product

$comp$

The product's company code (for internal use)

$prod$

The product's product code (for internal use)

$rowEvenOdd$

Will return a 0 or 1 for even or odd rows, (useful for css programming)

Note that $PremiumAnnual$ will always show the annual premium, whereas $Premium$ will show the premium with the mode you are using. By using these two together you can show the Annual and monthly premiums side by side, for example. Just be aware that $premium$ can also show the annual premium, so it is possible to have two columns of annual premiums.

Let's go through a very simple example:

$prodline$
Company Name is $company$ Premium is $premium$
$/prodline$

In this example, for every product line the output will be: Company Name is CompanyA Premium is 500.00

Of course, most of you would want to format the comparison results within a table of your own design. Take a look at the prodline definition in the sample TEMPLATE.HTM. It shows a very good example of how to define a prodline definition.
